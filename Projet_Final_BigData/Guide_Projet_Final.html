Introduction au sujet

Bienvenue ! Ce guide est conçu pour un débutant. Vous allez construire un système Big Data complet, capable de :

  • Recevoir en temps réel des données venant de milliers de compteurs intelligents (Linky, Smart Meters)
  • Les traiter en streaming avec Spark (agrégation par zone, par 15 min)
  • Les stocker dans une base NoSQL pour la vélocité et la haute disponibilité
  • Les analyser avec 3 modèles d'IA pour prédire la charge réseau
  • Afficher un dashboard interactif avec cartes de chaleur, courbes, alertes
Objectif final : permettre aux gestionnaires d'énergie d'anticiper les pannes par surcharge et d'optimiser l'intégration des énergies renouvelables (solaire, éolien) qui sont intermittentes.

Pourquoi Big Data ?

Imaginez 10 000 compteurs envoyant chacun une mesure toutes les 10 secondes. Cela représente 1 000 mesures/seconde, soit 86 millions de lignes par jour. Une base SQL classique s'effondre. C'est pourquoi on utilise des outils Big Data : Kafka (encaisser le flux), Spark (calculer en parallèle), MongoDB (stockage rapide).

  3. En bas à droite : Virtualisation : Activée

11.1 — Producteur compteurs (producer_meters.py)

import json, random, time
from datetime import datetime
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers='master:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

zones = ['Z1','Z2','Z3','Z4','Z5']

while True:
    meter = {
        "meter_id": f"M{random.randint(1, 10000):05d}",
        "zone": random.choice(zones),
        "timestamp": datetime.utcnow().isoformat(),
        "consumption_kwh": round(random.uniform(0.05, 5.0), 3),
        "voltage_v": round(random.gauss(230, 5), 2),
        "current_a": round(random.uniform(0.5, 25), 2),
    }
    producer.send('meters-raw', meter)
    print("Sent:", meter)
    time.sleep(0.1)              # 10 messages / seconde

12Étape B — Traitement Spark Structured Streaming

Spark va lire Kafka, agréger par zone sur des fenêtres de 15 minutes, puis écrire vers MongoDB.

12.1 — Script spark_streaming.py

from pyspark.sql import SparkSession
from pyspark.sql.functions import from_json, col, window, avg, sum as _sum, count
from pyspark.sql.types import StructType, StringType, DoubleType, TimestampType

spark = (SparkSession.builder
    .appName("EnergyStreaming")
    .config("spark.jars.packages",
            "org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.0,"
            "org.mongodb.spark:mongo-spark-connector_2.12:10.2.0")
    .config("spark.mongodb.write.connection.uri",
            "mongodb://master:27017/bigdata_energy")
    .getOrCreate())

schema = (StructType()
  .add("meter_id", StringType())
  .add("zone", StringType())
  .add("timestamp", TimestampType())
  .add("consumption_kwh", DoubleType())
  .add("voltage_v", DoubleType())
  .add("current_a", DoubleType()))

raw = (spark.readStream.format("kafka")
       .option("kafka.bootstrap.servers", "master:9092")
       .option("subscribe", "meters-raw")
       .option("startingOffsets", "latest")
       .load())

parsed = (raw.selectExpr("CAST(value AS STRING) as json")
          .select(from_json(col("json"), schema).alias("d"))
          .select("d.*"))

agg = (parsed
   .withWatermark("timestamp", "20 minutes")
   .groupBy(window(col("timestamp"), "15 minutes"), col("zone"))
   .agg(
       _sum("consumption_kwh").alias("total_kwh"),
       avg("voltage_v").alias("avg_voltage"),
       count("*").alias("nb_meters")
   ))

query = (agg.writeStream
   .format("mongodb")
   .option("checkpointLocation", "/tmp/spark-checkpoints/agg")
   .option("database", "bigdata_energy")
   .option("collection", "aggregations")
   .outputMode("append")
   .start())

query.awaitTermination()

12.2 — Lancer le job Spark

from textblob import TextBlob
from pyspark.sql.functions import udf
from pyspark.sql.types import DoubleType

@udf(DoubleType())
def sentiment_polarity(text):
    if text is None: return 0.0
    return float(TextBlob(text).sentiment.polarity)

# Exemple : tickets dans un topic Kafka 'tickets'
tickets_with_sentiment = tickets_df.withColumn(
    "polarity", sentiment_polarity(col("description"))

14.2 — Modèle 1 : ARIMA

from statsmodels.tsa.arima.model import ARIMA
from sklearn.metrics import mean_absolute_error

train = df["kwh"][:-100]
test  = df["kwh"][-100:]

model = ARIMA(train, order=(2,1,2)).fit()
pred  = model.forecast(steps=100)
mae_arima = mean_absolute_error(test, pred)

14.4 — Modèle 3 : LSTM

import numpy as np
from sklearn.preprocessing import MinMaxScaler
from tensorflow.keras.models import Sequential
from tensorflow.keras.layers import LSTM, Dense, Dropout

scaler = MinMaxScaler()
data = scaler.fit_transform(df[["kwh"]].values)

def make_seq(arr, n=24):
    X, y = [], []
    for i in range(len(arr)-n):
        X.append(arr[i:i+n])
        y.append(arr[i+n])
    return np.array(X), np.array(y)

X, y = make_seq(data, 24)
split = int(len(X)*0.8)
Xtr, Xte, ytr, yte = X[:split], X[split:], y[:split], y[split:]

m = Sequential([
    LSTM(64, return_sequences=True, input_shape=(24,1)),
    Dropout(0.2),
    LSTM(32),
    Dense(1)
])
m.compile(optimizer="adam", loss="mse")
m.fit(Xtr, ytr, epochs=15, batch_size=32, validation_split=0.1, verbose=0)

pred_lstm = scaler.inverse_transform(m.predict(Xte))
ytrue = scaler.inverse_transform(yte)
mae_lstm = np.mean(np.abs(pred_lstm.flatten() - ytrue.flatten()))
print(f"LSTM MAE={mae_lstm:.3f}")

20.2 — Questions probables (anticipation)

Pourquoi Kafka et pas RabbitMQ ? Kafka est conçu pour des débits massifs (millions msg/s), persiste les messages sur disque, supporte le replay, et s'intègre nativement avec Spark Structured Streaming. RabbitMQ est meilleur pour des messages transactionnels avec priorités et routage complexe — pas notre besoin.
Pourquoi Random Forest plutôt que LSTM en prod ? Sur notre dataset, RF obtient un MAE inférieur (2.85 vs 3.10) et un temps d'inférence 13× plus rapide (3 ms vs 40 ms), tout en restant interprétable (feature importance). Le LSTM aurait du sens si on avait des séquences très longues et des patterns non-linéaires complexes.
Que se passe-t-il si Kafka tombe ? Les producteurs Python ont un buffer + retry. En production réelle, on aurait 3 brokers Kafka (cluster) avec replication-factor=3 pour assurer la continuité. Spark a un mécanisme de checkpointing qui permet de reprendre là où il s'est arrêté.
Comment passer à 1 million de compteurs ? Augmenter les partitions Kafka (de 3 à 30+), augmenter les workers Spark, sharder MongoDB par zone géographique, déployer sur Kubernetes avec auto-scaling.
